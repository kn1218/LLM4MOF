# LLM2POR 프로젝트 파일 분석 (260324)

이 문서는 LLM2POR 프로젝트의 각 파일에 대해 분석하고 설명하는 문서입니다.

---

## 1. 프로젝트 개요

**LLM2POR**는 Large Language Model을 활용하여 Metal-Organic Framework (MOF)를 설계하는 자율 에이전트 시스템입니다. 이 시스템은 GPT/Gemini 같은 LLM을 사용하여 MOF设计方案을 제안하고, QMOF, hMOF, PORMAKE 같은 실제 컴퓨팅 데이터베이스에 대해 평가합니다.

### 작동 원리

```
사용자 Inquiry ("band gap 3-4 eV인 MOF를 설계해줘")
        │
        ▼
  [Agent 0] 문제 컨설턴트 (선택 사항)
    인터뷰를 통해 요구사항 명확화
        │
        ▼
  [Agent 1] 가설 생성기
    MOF 설계 제안 (금속, 리커, 기하학)
        │
        ▼
  [Agent 2] 제약 추출기
    가설을 검색 가능한 DB 필터로 변환
        │
        ▼
  [Matchmaker] 컴포넌트 발견
    DB에서 매칭되는 MOF 찾기
        │
        ▼
  [Sensitivity Analyzer] 평가
    가설의 성능 측정
        │
        ▼
  [Feedback Generator] 학습 시그널
    Agent 1을 위한 구조화된 피드백 생성
        │
        ▼
  Agent 1이 가설 개선 (반복)
```

## 3. 파일별 상세 분석

### 3.1 run_experiment.py (메인 진입점)

**역할**: 전체 자율 MOF 설계 루프를 조율하는 메인 엔트리 포인트

**주요 기능**:
1. ** modes 선택**: Agent 0 사용 여부, 직접 inquiry 선택
2. **指标 감지**: 사용자 입력에서 bandgap, H2, CH4, CO2 등의指标 자동 감지
3. **반복 루프**: Agent 1 → Agent 2 → Matchmaker → Sensitivity Analyzer → Feedback Generator
4. **파일 저장**: 각 반복마다 agent1_output.json, agent2_output.json, feedback_selected.txt 저장
5. ** экспеIMENT 디렉토리 관리**: `exp_YYYYMMDD_HHMM_{mode}/` 형식으로 결과 저장

**사용 방법**:
```bash
python run_experiment.py
```

### 3.2 config.py (설정)

**역할**: 모든 설정값과 데이터 경로를集中的으로 관리

**주요 섹션**:
1. **API 설정**: LLM_PROVIDER, OPENAI_API_KEY, GEMINI_API_KEY, 모델 선택
2. **경로 설정**: BASE_DIR, DATA_DIR, 각 데이터베이스 파일 경로
3. **기능 매니페스트**: 사용 가능한 데이터셋과 속성
4. **指标 레지스트리**: bandgap, H2 uptake, CH4 storage 등 매핑
5. **실험 설정**: FEEDBACK_SAMPLE_SIZE, STOCHASTIC_SAMPLING 등
6. **LLM 클라이언트 설정**: LLM_MAX_OUTPUT_TOKENS, 타임아웃 등

**핵심 상수**:
- `ACTIVE_METRIC_COLUMN`: 현재 최적화 대상 (기본값: "target" = H2 Storage)
- `CAPABILITY_MANIFEST`: Agent 0에게 시스템 기능 안내

### 3.3 core/llm_client.py (LLM 클라이언트)

**역할**: OpenAI와 Gemini API를 통일된 인터페이스로 지원

**주요 클래스**: `LLMClient`

**주요 기능**:
1. ** proveedor 지원**: OpenAI (GPT) 및 Google GeminiREST API 지원
2. **대화 기록 관리**: multi_turn=True로 Agent 1의 다중 턴 대화 지원
3. **JSON 추출**: `extract_json()` - LLM 응답에서 JSON 추출 (4가지 전략)
   - 전체 텍스트 JSON 파싱
   - 마크다운 코드 블록에서 찾기
   - 중괄호 패턴 매칭
   - 손상된 JSON 복구
4. **에러 처리**: 토큰 제한 시 경고 메시지, 재시도 로직

**테스트 함수**: `test_llm_client()` - API 키 필요

**역할**: Agent 0은 사용자와的多重对话를 통해 구조화된 문제 사양(JSON)을 생성

**주요 클래스**: `Agent0Handler`

**주요 기능**:
1. **시스템 프롬프트 로드**: `prompts/agent0_v3.md` 사용

3. **대화 상태 추적**: conversation_complete, problem_spec
4. **跳过 명령 처리**: 'proceed', 'skip', 'done', 'go' 입력 시 분석 진행
5. **문제 사양 변환**: `get_user_inquiry_text()` - JSON을 포맷된 텍스트로 변환

**출력 예시**:
```json
{
  "problem_specification": {
    "domain_category": "gas_storage",
    "target_molecules": ["H2"],
    "operating_conditions": {"temperature": "77K", "pressure": "100 bar"}
  }
}
```

### 3.5 core/agent1_handler.py (가설 생성기)

**역할**: Agent 1은 MOF 설계 가설을 생성하고 피드백을 통해 개선

**주요 클래스**: `Agent1Handler`

**주요 기능**:
1. **초기 가설 생성**: `generate_initial_hypothesis()` - 사용자 inquiry 기반
2. **가설 정제**: `refine_hypothesis()` - 피드백 기반 반복 개선
3. **과학 저널 주입**: `_update_system_prompt_with_journal()` - 이전 반복 정보 프롬프트에 포함
4. **대화 기록 관리**: 다중 턴 모드로 이전 맥락 유지

**출력 JSON 필드** (예시):
- `target_application`: 목표 애플리케이션
- `hypothesis_mechanism`:工作机制
- `node_composition`: 노드 구성 (금속, 클러스터 유형)
- `linker_composition`: 리커 구성
- `ideal_pore_geometry`: 기하학적 특성 (Di, Df, SA, VF)
- `lesson_learnt`: 이전 반복에서의 교훈

**테스트 함수**: `test_agent1()` - API 키 필요

### 3.6 core/agent2_handler.py (제약 추출기)

**역할**: Agent 1의 定性的 가설을 定量的 데이터베이스 검색 사양으로 변환

**주요 클래스**: `Agent2Handler`

**주요 기능**:
1. **제약 추출**: `extract_constraints()` - 가설 JSON → 검색 필터

3. **구조 검증**: V3 스키마 필수 키 확인 (node_query, linker_query, geometry_filter)

### 3.7 core/matchmaker.py (PORMAKE 매치메이커)

**역할**: Agent 2의 제약을 기반으로 PORMAKE 데이터베이스에서 유효한 토폴로지, 노드, 리커 발견

**주요 클래스**: `Matchmaker`

**주요 기능**:
1. **토폴로지 발견**: `get_valid_topologies()` - 단일 노드/에지 토폴로지만 선택
2. **노드 검색**: `_search_nodes()` - 금속, 연결성,核性 필터링
3. **리커 검색**: `_search_linkers()` - 길이, 강성, 기능기 필터링
4. **통합 로직**: `check_global_requirements()` - 노드+리커 조합 유효성 검사
5. **추상 기능 필터**: `abstract_features` (예: is_conjugated, has_open_metal_site)

**Phase별 처리**:
- Phase A: 토폴로지 발견 (전체 CN 리스트 사용)
- Phase B: 노드 검색 (CN별 루프)
- Phase C: 리커 검색 (노드 CN과 독립적)
- Phase D: 가상 조립 및 통합 로직
- Phase E: 분류된 기능기 필터 (backbone_requirements, substituent_requirements)

### 3.8 core/qmof_matchmaker.py (QMOF 매치메이커)

**역할**: QMOF 데이터베이스에서 bandgap 최적화를 위한 직접 MOF 레벨 필터링

**주요 클래스**: `QMOFMatchmaker`

**주요 기능**:
1. **금속 검사**: `_check_metals()` - OR 로직 (하나라도 일치)
2. **연결성 검사**: `_check_connectivity()` - 노드 연결점 수
3. **AND 태그 검사**: `_check_and_tags()` - 모든 태그 있어야 함
4. **OR 태그 검사**: `_check_or_tags()` - 하나라도 있으면 통과
5. **전자 메타데이터 필터** (Phase 3):
   - 산화 상태 (`oxidation_state`)
   - 기하학 선호도 (`geometry_preference`)
   - 개방 금속site (`has_open_metal_site`)
   - 토폴로지 (`topology`)
6. **데이터 검색**: `get_bandgap_data()` - 매칭된 QMOF의 bandgap 데이터 반환

**사용 시나리오**: "band gap 3-4 eV인 MOF" 등 전자 특성 최적화

---

### 3.9 core/hmof_matchmaker.py (hMOF 매치메이커)

**역할**: hMOF 데이터베이스에서 gas adsorption (H2, CH4, CO2, Xe/Kr) 최적화

**주요 클래스**: `HMOFMatchmaker`

**주요 기능**:
1. ** gas 속성**: H2 uptake (2bar, 100bar), CH4, CO2, Xe/Kr 선택성
2. **구조 속성**: surface_area, void_fraction, density, PLD, LCD
3. **범위 검사**: `_check_range()` - min/max 값으로 필터링
4. **구조 필터**: `structural_filters` 파라미터로 속성별 범위 지정
5. **데이터 검색**: `get_gas_data()`, `get_structural_data()`

**데이터베이스**: 51,163개의 가설 MOF (Snurr 그룹의 GCMC 시뮬레이션)

**사용 시나리오**: "고압 H2 저장", "메탄 저장", "CO2 포집" 등

### 3.10 core/sensitivity_analyzer.py (민감도 분석기)

**역할**: 필터링된 MOF 후보에 대한 성능 분석 (Enrichment Factor, P-Value 계산)

**주요 클래스**: `SensitivityAnalyzer`

**주요 기능**:
1. **필터 세트 생성**: 다양한 조합으로 데이터 분할
   - Set A: 화학만 (금속 + 리커)
   - Set D: 화학 + Di
   - Set E: Di + Df
   - Set Z: 완전한 가설 (화학 + Di + Df)
2. **Enrichment Factor 계산**: 상위 1%, 5%에서 성능 향상 측정
3. **통계 검정**: Mann-Whitney U-test로 P-value 계산
4. **QMOF 모드**: bandgap 범주별 분류 (Metallic, Narrow/IR, Vis, UV, Insulator)
5. **hMOF 모드**: gas adsorption 속성별 분석

**중요**: 이 분석은 **인간 전용**으로, Agent 1과 공유되지 않음

---

### 3.11 core/feedback_generator.py (피드백 생성기)

**역할**: Agent 1을 위한 과학적으로 설계된 7가지 피드백 타입 생성

**피드백 타입** (정보 가치 순위):

1. **3-Beam Diagnostic** (최고 가치):
   - Beam 1: 완전한 가설
   - Beam 2: 화학만 (무작위 기하학)
   - Beam 3: 기하학만 (무작위 금속)

2. **Universe Baseline**: 전체 DB에서 무작위 샘플링

3. **Geometric Optimizer**: A/B 테스트 (무작위 vs 제한된 기하학)

4. **Chemical Pivot**: A/B 테스트 (내 금속 vs 모든 금속)

5. **Best vs Worst**: 상위 15개 vs 하위 15개 비교

6. **Hypothesis Validation**: 완전한 가설만 테스트

7. **Virtual Synthesis**: 전향합성 시뮬레이션

**주요 기능**:
- `_generate_chemistry_profile()`: 각 샘플의 추상 기능과 분류된 기능기
- `_generate_pattern_summary()`: 집계 통계 (500자 제한)
- `_generate_diagnostic_footer()`: 0개 결과 시 실패 원인 설명

### 3.13 core/name_resolver.py (이름 해석기)

**역할**: 빌딩 블록 ID를 사람이 읽을 수 있는 이름으로 변환

**주요 클래스**: `NameResolver` (Singleton 패턴)

**주요 기능**:
1. **ID → 이름 매핑**: "N164" → "Zr6 Cluster"
2. **MOF 파일명 변환**: "ukd+N164+E70" → "Node: Zr6 Cluster | Linker: BDC"
3. ** singlet ON 패턴**: `get_name_resolver()` - 메모리 절약

**사용 이유**: AI가 의미 없는 알파벳-숫자 코드를 분석하여 화학 학습 불가

### 3.14 core/constraint_utils.py (제약 유틸리티)

**역할**: Agent 2의 화학 제약을 파싱하고 적용하는 중앙 집합소

**주요 함수**:
1. **canon()**: 정규화 (소문자, 언더스코어, 别名 해결)
2. **parse_functional_groups()**: AND/OR/Negative 태그 분리
3. **get_approved_vocab()**: 승인된 온톨로지 태그 집합 반환
4. **check_negative_tags()**: 금지된 태그 검사
5. **check_global_requirements()**: 노드+리커 조합 유효성
6. **check_categorized_groups()**: backbone/substituent 분류 검증

**핵심 개념**:
- **AND 태그**: 모두 존재해야 함 (전역 요구사항)
- **OR 태그**: 하나라도 존재 (링커 기능기)
- **Negative 태그**: 포함하면 제외

## 5. 실행 흐름 상세

### 5.1 실험 시작
1. `python run_experiment.py` 실행
2. 모드 선택: Agent 0 사용 or 직접 inquiry
3. 지표 감지: QMOF/hMOF/PORMAKE 모드 자동 선택

### 5.2 반복 루프 (Iteration N)
1. **Agent 1**: 가설 생성 또는 정제
2. **Agent 2**: 가설 → 검색 필터 변환
3. **Matchmaker**: 필터 → DB 매칭
4. **Sensitivity Analyzer**: 결과 분석
5. **Feedback Generator**: 피드백 생성

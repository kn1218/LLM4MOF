# Supplementary Information: Database Construction Methodology

## Overview

This document describes the complete workflow for constructing the structured metadata databases used in LLM4MOF. Three MOF databases (PORMAKE, QMOF, hMOF) were processed through a two-layer enrichment pipeline to produce machine-readable JSON records suitable for LLM-driven screening.

Every record follows a universal two-layer architecture:

- **Layer 1 (Deterministic Facts):** Computed by Python + RDKit from source data. Verifiable, reproducible, no LLM involvement. Includes formula, topology, pore geometry, gas adsorption, bandgap, metal node characterization, and SMILES representations.
- **Layer 2 (Semantic Enrichment):** Generated by SMARTS pattern matching (105 curated patterns via RDKit) and LLM enrichment (GPT-4o-mini or Gemini 2.0 Flash). Includes functional group classification, human-readable names, design hints, and abstract feature flags.

### Summary Statistics

| Database | Records | Source Format | Raw Material Source | Layer 2 LLM | SMARTS Patterns |
|----------|---------|--------------|---------------------|-------------|-----------------|
| PORMAKE | 867 building blocks | Extended XYZ + bond block | PORMAKE library (KAIST) | Gemini 2.0 Flash | 105 |
| QMOF | 20,373 MOFs | CSV + MOFid decomposition | Rosen et al., *Matter* (2021) | GPT-4o-mini | 105 |
| hMOF | 51,163 MOFs | JSONL + MOFid decomposition | Wilmer et al., *Nat. Chem.* (2012) | GPT-4o-mini | 105 |

## 1. PORMAKE Building Block Pipeline (867 entries)

### 1.1 Raw Materials

**Source:** Extended XYZ files (`.xyz`) from the PORMAKE library (Sangwon Lee, Jihan Kim group, KAIST).

Each XYZ file encodes a molecular building block fragment used to computationally assemble MOFs. Building blocks are classified as:
- **Edges** (219 files, prefix `E`): Organic linkers with exactly 2 connection points
- **Nodes** (648 files, prefix `N`): Metal clusters or organic hubs with 3+ connection points

**XYZ file format:**
```
<atom_count>
<comment line (building block ID)>
<element> <x> <y> <z>          # one line per atom (Angstroms)
...
<atom_i> <atom_j> <bond_type>  # explicit bond block (S/D/T/A)
...
```

Key features of these XYZ files that required special handling:
- **Dummy atoms** (`X`): Terminal placeholder atoms marking where the building block attaches to the MOF framework
- **Aromatic bond type** (`A`): PORMAKE encodes resonance-delocalized bonds (nitro, carboxylate, sulfonate) as aromatic, which breaks standard SMARTS patterns

**Representative samples:** See `1_PORMAKE_pipeline/source_samples/` (E1.xyz, N1.xyz, N109.xyz)

### 1.2 Layer 1: Deterministic Feature Extraction

All quantitative properties are computed directly from XYZ files using Python, RDKit, and NumPy. No LLM involvement.

**Scripts (executed in order):**

| Step | Script | Input | Output |
|------|--------|-------|--------|
| 0 | `00_config.py` | -- | Constants, paths, METAL_SET, ATOMIC_WEIGHTS |
| 1 | `01_xyz_parser.py` | `.xyz` file | Parsed atoms, bonds, dummy atom positions |
| 2 | `02_feature_extractor.py` | Parsed XYZ | Formula, MW, atom counts, metals, bond graph, geometry, planarity, rigidity |
| 3 | `03_smiles_generator.py` | Parsed XYZ + RDKit | SMILES, canonical SMILES, SELFIES |

#### 1.2.1 XYZ Parsing (`01_xyz_parser.py`)

The parser reads each XYZ file line-by-line:
1. Line 1: atom count
2. Line 2: comment/ID
3. Lines 3 to N: element symbol + Cartesian coordinates (x, y, z)
4. Lines N+1 onward: bond block entries as `atom_i atom_j bond_type`

Special handling:
- Handles 11 files with blank comment headers
- Deduplicates reversed bond entries (e.g., `3-5` and `5-3`) affecting 81 files
- Dummy atoms (`X`) are tracked separately for connection point analysis

#### 1.2.2 Feature Extraction (`02_feature_extractor.py`)

From the parsed XYZ data, the following properties are computed:

- **Formula:** Hill-order chemical formula (C first, H second, then alphabetical), excluding dummy atoms
- **Molecular weight:** Summed from a hardcoded atomic mass table (~100 elements)
- **Metals:** Each element checked against `METAL_SET`; metal atoms collected into a list
- **Nuclearity:** Count of metal atoms (nodes only)
- **Connection points:** Count and 3D coordinates of dummy atoms
- **Geometry inference (NumPy SVD):**
  - 2 connection points: "linear"
  - 3+ connection points: SVD on centered coordinates to find plane of best fit (PBF)
    - PBF deviation < 0.2 A for 4 points: "square_planar"
    - PBF deviation > 0.5 A for 4 points: "tetrahedral"
    - Otherwise: "{N}-coordinate"
- **Length:** Maximum Euclidean distance between dummy atoms (edges only)
- **Bond graph:** Bond count by type (S/D/T/A), ring count, ring sizes (via NetworkX cycle detection)
- **Rigidity:** Bridge-based analysis (bonds whose removal disconnects the graph), filtered by degree > 2
- **Planarity:** Assessed from 3D coordinate spread perpendicular to PBF

#### 1.2.3 SMILES Generation (`03_smiles_generator.py`)

Two methods are attempted in order:

**Phase 1 -- Explicit Bond Parsing (primary, used for 867/867 files):**
1. Parse bond block from XYZ into `(atom_i, atom_j, bond_type)` tuples
2. Create RDKit editable molecule (`Chem.RWMol()`)
3. Add atoms; dummy atoms `X` temporarily represented as **Lawrencium (Lr, Z=103)** because RDKit's wildcard `*` causes valence errors during construction
4. Add bonds using type mapping: S -> SINGLE, D -> DOUBLE, T -> TRIPLE, A -> AROMATIC
5. Flag aromatic atoms explicitly
6. Sanitize with **kekulization disabled** (building block fragments have dangling aromatic bonds)
7. Replace Lr back to dummy atom (atomic number 0 = `*` in SMILES)
8. Generate canonical SMILES via `Chem.MolToSmiles(mol, canonical=True)`

**Phase 2 -- xyz2mol Geometry Fallback (when bond block is missing):**
1. `xyz2mol` (Jensen group, open-source) infers bonds from 3D interatomic distances (covalent radii + tolerance)
2. Bond orders assigned via Huckel method
3. Resulting molecule converted to SMILES

**Note:** Phase 2 was never triggered for the current dataset. All 867 PORMAKE XYZ files contain explicit bond blocks, so Phase 1 succeeded for every building block. The xyz2mol fallback is included for completeness (`03b_xyz2mol.py`).

**Validation -- Degree of Unsaturation (DoU) Check:**
- DoU calculated from formula: `DoU = C + 1 - H/2 + N/2 - halogens/2`
- If DoU >= 4 but SMILES has no unsaturated bonds, the SMILES is rejected (catches xyz2mol failure mode where aromatics appear as saturated cages)

**SELFIES:** Generated from valid SMILES using the `selfies` library; returns `null` for metal-containing molecules (not supported).

### 1.3 Layer 2: Semantic Enrichment

#### 1.3.1 SMARTS Pattern Matching (`04_smarts_library.py` + `05_smarts_enricher.py`)

105 curated SMARTS patterns are matched against each building block's SMILES using RDKit's `HasSubstructMatch()`. Patterns are organized into four categories that determine classification:

| Category | Count | Classification | Examples |
|----------|-------|---------------|----------|
| Scaffolds | 13 | backbone | benzene_ring, biphenyl, naphthalene, pyrene, porphyrin |
| Heterocycles | 25+ | backbone | pyridine, imidazole, triazole, thiophene, furan |
| Functional groups | 50+ | backbone | carboxylate, amine, hydroxyl, nitro, ether, imine, azo |
| Substituents | 12 | substituents | methyl, methoxy, trifluoromethyl, fluorine, chlorine |

**Critical fix for aromatic bond encoding:** PORMAKE XYZ files encode resonance-delocalized bonds (nitro, carboxylate, sulfonate, phosphonate) as type `A` (aromatic). Standard SMARTS patterns that specify explicit single/double bonds silently fail. The solution uses bond-order-agnostic SMARTS alternatives:
- Nitro: `[#7]([#8])([#8])` instead of `[N+](=O)[O-]`
- Carboxylate: `[#6X3](~[#8])(~[#8])` instead of `[CX3](=O)[O-]`

This fix increased carboxylate detection from 2 to 202 building blocks.

**Metal-stripping fallback:** For 553 metal-containing BBs where metals interfere with SMARTS matching, metals are stripped from the SMILES and functional groups are matched on the remaining organic fragments. This increased enrichment from 667 to 808 BBs.

**Multiplicity counting:** Each pattern reports match count (e.g., `benzene_ring: 3` for terphenyl). Stored in `rule_based_counts`.

**Generic tag propagation:** After matching, parent tags are automatically added (e.g., `benzene_ring` adds `aromatic`, `aryl`, `ring`).

#### 1.3.2 Node Enrichment (`06_node_enricher.py`)

Deterministic enrichment for metal-containing nodes:
- **SBU type:** Standardized label combining metal, geometry, and connectivity (e.g., `Zn4_O_octahedral_6c`)
- **Ligand chemistry:** Donor atom types identified from `metal_coordination` (e.g., `["O"]` for carboxylate-coordinated nodes)

#### 1.3.3 LLM Enrichment (`07_llm_enricher.py`)

After SMARTS-based tagging, Gemini 2.0 Flash generates:
- **readable_name:** Human-readable building block name (e.g., "pyrene-based 4-connected node")
- **design_hints:** Brief text on potential MOF design applications
- **abstract_features:** Boolean flags (is_fluorinated, is_electron_rich, is_conjugated, is_symmetric, is_metalated, has_hydrogen_bond_donor, has_hydrogen_bond_acceptor, is_charged, is_photoswitchable)
- **llm_additions:** Functional groups identified by LLM but missed by SMARTS

Settings: temperature = 0.3 (near-deterministic), JSON structured output enforced.

**Full LLM prompt:** The complete system prompt is embedded in `07_llm_enricher.py` (lines 20-36). It instructs the LLM to return a JSON object with three fields: `readable_name` (2-8 word chemical name), `design_hints` (1-2 sentences on MOF design potential), and `llm_additions` (functional groups SMARTS may have missed). The prompt includes a curated list of common SMARTS blind spots (porphyrin, paddlewheel, IRMOF-type, UiO-type, catechol, salen, etc.) and rules to avoid duplicating rule_based tags.

### 1.4 Validation (`10_validator.py`)

14 automated checks per building block:
- Formula non-empty, molecular weight > 0
- Connection points present, edges have exactly 2
- SMILES contains `[*]` wildcards
- Bond graph is connected
- Edge length > 0

Results: 866 pass, 1 warning (N428: disconnected graph), 0 failures.

### 1.5 Master Pipeline (`08_pipeline.py`)

Orchestrates all phases: XYZ parsing -> feature extraction -> SMILES generation -> SMARTS enrichment -> node enrichment -> LLM enrichment -> validation. Processes all 867 building blocks, outputs individual JSON files to `bb_metadata_v8/`.

### 1.6 Aggregation (`11_build_pormake_bb_v5.py`)

Reads 867 individual JSON files and converts them into a single flat array (`pormake_bb_dictionary_v5.json`) in the format expected by the LLM4MOF screening engine. Key transformations:
- 4-layer JSON schema flattened to single-level fields
- `connection_chemistry` element symbols mapped to full names (e.g., `"C"` -> `"Carbon"`)
- `ligand_chemistry` element symbols mapped similarly
- New fields added: `functional_groups_categorized` (backbone/substituents split), `abstract_features`, `design_hints`, `sbu_type`

### 1.7 Post-Processing (`12_enrich_generic_tags.py`)

Adds generic/abstract parent tags to each record's functional group list:
- e.g., `benzene_ring` -> also adds `aromatic`, `aryl`, `ring`
- Ensures vocabulary consistency with the unified vocabulary

## 2. QMOF Pipeline (20,373 entries)

### 2.1 Raw Materials

**Source:** QMOF database (Rosen et al., *Matter*, 2021. DOI: 10.1016/j.matt.2021.02.015)

Two input data sources:
1. **`qmof.csv`** (20,373 rows): DFT-computed properties including bandgaps at 4 functional levels (PBE, HLE17, HSE06-10HF, HSE06), structural descriptors (density, volume, PLD, LCD), space group, crystal system, DOI references
2. **`qmof_global_jsons_v2/`** (20,372 files): Pre-computed deterministic analysis of each MOF's metal nodes and organic linkers, generated from CIF crystal structures using MOFid decomposition + RDKit

All QMOF entries are experimentally synthesized MOFs (`synthesized = true`).

### 2.2 Layer 1: Deterministic Feature Extraction

**Scripts:**

| Step | Script | Purpose |
|------|--------|---------|
| 1 | `01_pipeline_v2.py` | Master pipeline: CSV parsing + integration |
| 2 | `02_schema_v2.py` | Two-layer JSON schema definition |
| 3 | `03_metal_enricher.py` | Metal node characterization from node SMILES |
| 3b | `03b_source_investigator.py` | Source data quality investigation |

#### 2.2.1 CSV Extraction

DFT bandgaps (4 levels), structural descriptors (density, PLD, LCD, volume), space group, and crystal system are read directly from `qmof.csv`.

#### 2.2.2 MOFid Decomposition

Each MOF's MOFid string is parsed to extract:
- **Topology** (e.g., pcu, dia, sql) -- 10,373 of 20,373 have valid topology
- **SMILES for metal nodes** (e.g., `[Zn][O]([Zn])([Zn])[Zn]`)
- **SMILES for organic linkers** (e.g., `[O-]C(=O)c1ccc(cc1)C(=O)[O-]`)
- SMILES validity flags (parsed by RDKit)

#### 2.2.3 Metal Node Analysis (`03_metal_enricher.py`)

From node SMILES and structural context:
- **Metals:** Element symbols extracted from SMILES
- **Nuclearity:** Metal atom count per node
- **Oxidation states:** Deduced from ligand environment (e.g., `{"Cu": 2}`)
- **Spin state:** Inferred from d-electron configuration (e.g., "Doublet (d9)")
- **Coordinating groups:** Identified from ligand SMILES (Carboxylate, Nitrogen, etc.)
- **Connectivity and geometry:** Inferred from coordination number

### 2.3 Layer 2: Semantic Enrichment

#### 2.3.1 SMARTS Pattern Matching (`04_smarts_engine.py`)

The same 105 SMARTS patterns from `5_shared/smarts_library.py` are applied to each linker's SMILES:
- `HasSubstructMatch()` identifies functional groups
- Results classified into backbone vs substituents
- Multiplicity counted (`rule_based_counts`)

#### 2.3.2 LLM Enrichment (`05_llm_enricher.py`)

GPT-4o-mini generates `readable_name`, `design_hints`, and `abstract_features` for each MOF record. Settings: temperature = 0.3, JSON structured output enforced.

**Full LLM prompt:** The complete system prompt is embedded in `05_llm_enricher.py` (lines 89-134). A shared base prompt requests the same three fields as PORMAKE (`readable_name`, `design_hints`, `llm_additions`), with database-specific context injected: QMOF records are described as "real experimentally characterized MOFs with DFT-computed bandgap", hMOF records as "HYPOTHETICAL (computationally generated) MOFs... NOT been synthesized", and CoRE-MOF records as "real experimentally synthesized MOFs with stability data".

Batch processing with checkpoint/resume: enrichment runs in batches of 500, with progress saved to checkpoint files. Safe to interrupt and restart. Total enrichment time: ~13 hours at ~2.3s/record. Cost: ~$4.60 (GPT-4o-mini).

### 2.4 Validation (`06_validator_v2.py`)

Automated quality checks per record. Results: 5,489 pass, 14,883 warnings (mostly missing topology from MOFid decomposition failures), 0 failures.

### 2.5 Aggregation (`07_build_qmof_index_v2.py`)

Reads 20,373 individual JSON files and produces `qmof_index_v2.json` -- a monolithic JSON array in the flat format expected by LLM4MOF.

### 2.6 Post-Processing (`08_enrich_qmof_tags.py`)

Adds generic/abstract parent tags for consistency with the unified vocabulary.

### 3.1 Raw Materials

- Only 4 metals used: Zn (62%), Cu (26%), V (7%), Zr (5%)

Same as QMOF:
- 105 SMARTS patterns applied to linker SMILES

- Batch processing with checkpoint/resume. Total time: ~33 hours. Cost: ~$9.20.

Reads 51,163 individual JSONs and produces:
- `hmof_index.json` (compact, all records)
- `hmof_index_pretty_sample.json` (first 10 records, indented for inspection)

### 4.3 Metal Node Enrichment (`02_enrich_coremof_metal_nodes.py`)

## 5. Vocabulary Construction (Unified Vocabulary)


### 5.1 Purpose

The unified vocabulary (`unified_vocabulary.json`, v1.2) provides a canonical vocabulary of functional group tags that is consistent across all three databases. This vocabulary is used by the LLM screening engine (Agent 2) to translate natural-language chemical reasoning into typed database constraints.

### 5.2 SMARTS Pattern Library (`01_vocabulary_dictionary.py` + `5_shared/smarts_library.py`)

105 curated RDKit-compatible SMARTS patterns organized into 4 categories:

1. **Scaffolds (13 patterns):** Primary structural backbone identification
   - Polycyclic aromatics: pyrene, anthracene, naphthalene, fluorene, phenanthroline
   - Multi-ring linkers: biphenyl, terphenyl
   - Macrocycles: porphyrin, cubane, adamantane, triptycene
   - Saturated: cyclohexane, cyclopentane

2. **Heterocycles (25+ patterns):** Aromatic and non-aromatic heterocyclic rings
   - 6-membered N-heterocycles: pyridine, pyrimidine, pyrazine, triazine
   - 5-membered N-heterocycles: imidazole, pyrazole, triazole (1,2,3 and 1,2,4), tetrazole
   - S/O-heterocycles: thiophene, furan, thiazole, oxazole
   - Fused: benzimidazole, quinoline, indole

3. **Functional Groups (50+ patterns):** Core chemical moieties
   - Carboxyl/carboxylate, amines (primary/secondary/tertiary), amide, hydroxyl, phenol
   - Nitro, nitrile, thiol, sulfonate, phosphonate
   - Ether, ester, ketone, aldehyde, imine, azo

4. **Substituents (12 patterns):** Small pendant groups
   - Halogens: fluorine, chlorine, bromine, iodine
   - Alkyl: methyl, tert-butyl
   - Others: methoxy, trifluoromethyl, vinyl, acetylene, butadiyne

### 5.3 Backbone vs Substituent Classification Logic

The classification of detected functional groups into "backbone" vs "substituent" is **hardcoded by chemical role**, not determined by molecular position or bonding topology.

Each SMARTS pattern in `smarts_library.py` is defined as a frozen `SmartsPattern` dataclass with a `category` field:

```python
@dataclass(frozen=True)
class SmartsPattern:
    name: str        # e.g., "carboxylic_acid"
    smarts: str      # SMARTS string
    category: str    # "functional_group", "scaffold", "substituent", "heterocycle", "abstract"
    description: str
```

The enrichment engine (`smarts_enricher.py`) splits detected patterns into backbone and substituents based solely on this category:

```
backbone    = all matches where category is "functional_group" OR "scaffold" OR "heterocycle"
substituents = all matches where category is "substituent"
```

**The 13 substituent patterns** (the only patterns classified as substituents):

| Pattern | SMARTS | Description |
|---------|--------|-------------|
| methyl | `[CH3;!$(C=O)]` | Methyl group (not part of acetyl) |
| trifluoromethyl | `[CX4](F)(F)F` | CF3 group |
| methoxy | `[OX2][CH3]` | Methoxy group |
| tert_butyl | `[CX4]([CH3])([CH3])[CH3]` | tert-Butyl group |
| fluorine | `[FX1]` | Fluorine substituent |
| chlorine | `[ClX1]` | Chlorine substituent |
| bromine | `[BrX1]` | Bromine substituent |
| iodine | `[IX1]` | Iodine substituent |
| vinyl | `[CX3]=[CX3]` | Vinyl/alkene |
| acetylene | `[CX2]#[CX2]` | Alkyne |
| butadiyne | `[CX2]#[CX2][CX2]#[CX2]` | Diacetylene |
| nitrile_sub | `[CX2]#[NX1]` | Nitrile as substituent |
| nitro_sub | `[$([NX3](=O)=O)]` | Nitro as substituent |

Everything else (carboxylate, benzene_ring, pyridine, imidazole, porphyrin, etc.) is classified as **backbone** regardless of its position in the molecule.

**Why chemical role, not topology?** In the context of MOF building blocks, substituents are small pendant groups that decorate the main scaffold but do not participate in framework connectivity. This distinction is relevant for LLM-driven screening because it tells Agent 1 which chemical features define the structural backbone vs which are tunable decorations.

**Consistency across databases:** The QMOF SMARTS engine (`smarts_engine.py`) imports the same `smarts_library.py` and uses identical classification logic. There is no pattern duplication or divergence between PORMAKE and QMOF/hMOF processing.

### 5.4 Generic Tag Propagation

After SMARTS detection, a post-processing step (`12_enrich_generic_tags.py` for PORMAKE, `08_enrich_qmof_tags.py` for QMOF) adds generic/abstract parent tags derived from the specific detected tags. This ensures vocabulary consistency across databases and alignment with the unified vocabulary.

The mapping is defined in a hardcoded `TAG_HIERARCHY` dictionary (~152 entries). Representative mappings:

| Specific Tag | Generic Tags Added |
|-------------|-------------------|
| `benzene_ring` | `aromatic`, `aryl`, `ring` |
| `naphthalene` | `aromatic`, `aryl`, `ring` |
| `pyridine` | `heterocycle`, `ring`, `nitrogen`, `aromatic` |
| `imidazole` | `heterocycle`, `ring`, `nitrogen`, `aromatic`, `azole`, `azolate` |
| `thiophene` | `heterocycle`, `ring`, `sulfur`, `aromatic` |
| `carboxylic_acid` | `carbonyl` |
| `trifluoromethyl` | `halogen` |
| `fluorine` | `halogen` |

**Example:** If a building block has SMARTS-detected tags `["benzene_ring", "carboxylic_acid"]`, after generic tag propagation it becomes `["aromatic", "aryl", "benzene_ring", "carbonyl", "carboxylic_acid", "ring"]`.

### 5.5 Bridge Dictionary (`02_qmof_vocabulary_parser.py`)

Maps lowercase SMARTS tag names to title-case canonical vocabulary keys (e.g., `benzene_ring` -> `Benzene`, `carboxylic_acid` -> `Carboxylate`). Also reads the hierarchical edge vocabulary to build degenerate (multiple-inheritance) mappings where each tag maps to all its topological/compositional ancestors.

### 5.6 Cross-Database Tag Audit (`03_vocab_audit.py`)

Scans three metadata sources (PORMAKE, QMOF, hMOF) for functional-group tags, builds frequency distributions, and generates cross-reference reports with synonym detection. Uses a `canon()` normalization function (lowercase, strip whitespace, hyphens/spaces to underscores) for consistent comparison.

### 5.7 Vocabulary Assembly (`04_build_vocab_mapping.py`)

Consolidates tags from four input sources into a single canonical vocabulary:
1. **PORMAKE BB dictionary v3.2** (869 building blocks)
2. **v7 edge metadata** (222 enriched edge files)
3. **QMOF global JSONs** (20,372 MOF analyses)
4. **hMOF index** (51,163 MOF records)

Output: `unified_vocabulary.json` (v1.2) containing:
- 107 canonical tags in Title_Case
- ~300+ alias mappings (case-insensitive lookup)
- Per-tag metadata: category (backbone/scaffold/substituent/heterocycle/etc.), `agent2_approved` flag
- Separate sections for QMOF-only terminal ligand tags and metal symbols

### 5.8 Runtime Integration

At runtime, `constraint_utils.py` in the LLM4MOF screening engine loads `unified_vocabulary.json` to provide:

1. **Alias resolution:** When Agent 2 outputs a constraint like `"Aromatic_Ring"`, the vocabulary maps it to the canonical form `"Aromatic"`, which in turn matches the `"aromatic"` tag in the database records
2. **Vocabulary validation:** Ensures Agent 2 only uses tags that exist in the database, preventing hallucinated constraints
3. **Case-insensitive lookup:** All tags are normalized to lowercase before matching against the database

The canonical vocabulary convention is Title_Case (e.g., `Aromatic`, `Benzene`, `Pyridine`), while the database stores lowercase tags (e.g., `aromatic`, `benzene_ring`, `pyridine`). The vocabulary's alias system bridges this gap transparently.

## 6. Field Usage in the LLM4MOF screening pipeline

The enrichment pipeline generates comprehensive metadata for each record. However, not all fields are consumed by the LLM4MOF screening engine at runtime. This section documents which fields actively drive the screening workflow versus those that serve as enrichment metadata for human inspection or future extensions.

### 6.1 Fields Used in Constraint Filtering (Agent 2 -> Screening Engine)

These fields are read by the matchmaker and constraint validation logic to filter candidates. They directly affect screening results.

| Field | Source | Usage | Code Reference |
|-------|--------|-------|---------------|
| `functional_groups` | SMARTS (deterministic) | Core constraint matching -- every candidate is evaluated against FG constraints | `constraint_utils.py`, `matchmaker.py`, `hmof_matchmaker.py`, `qmof_matchmaker.py` |
| `ligand_chemistry` / `connection_chemistry` | Deterministic (from XYZ/MOFid) | Chemical constraint matching for linker donor atoms | `matchmaker.py`, `constraint_utils.py` |
| `nuclearity` | Deterministic (metal atom count) | Exact-match filtering for metal cluster size | `matchmaker.py:210-242` |
| `abstract_features` | SMARTS + LLM (boolean flags) | Boolean feature filtering (e.g., is_rigid, is_conjugated) | `matchmaker.py:148-155` |
| `connectivity` | Deterministic (connection point count) | Node/edge connectivity constraint | `matchmaker.py:199, 279` |
| `metals` | Deterministic (from formula/SMILES) | Metal element inclusion/exclusion | `matchmaker.py:214` |
| `is_rigid` | Deterministic (bridge analysis) | Linker rigidity constraint | `matchmaker.py:289` |
| `length` | Deterministic (dummy atom distance) | Linker length range filtering | `matchmaker.py:283` |

**Key observation:** All fields that drive filtering are either fully deterministic (Layer 1) or SMARTS-based (reproducible). The screening pipeline does not depend on non-reproducible LLM-generated content for its filtering decisions.

### 6.2 Fields Used in Feedback Generation (Shown to Agent 1)

These fields are displayed in the feedback loop to help Agent 1 understand the chemistry of sampled candidates. They do not affect filtering but influence Agent 1's reasoning.

| Field | Source | Usage |
|-------|--------|-------|
| `readable_name` | LLM-generated | Human-readable descriptions of candidates |
| `functional_groups_categorized` | SMARTS (deterministic) | Backbone/substituent breakdown in chemistry profile |
| `topology` | Deterministic (MOFid) | Displayed in pattern summaries |
| `geometry` | Deterministic (QMOF metal_enricher) | Appended to QMOF descriptions |
| `oxidation_states` | Deterministic (QMOF metal_enricher) | QMOF chemistry profile display |

### 6.3 Fields Not Used at Runtime (Enrichment Metadata Only)

These fields are generated during enrichment but are never read by the screening engine or feedback generator. They serve as metadata for human inspection, downstream analysis, or potential future use.

| Field | Source | Why Unused |
|-------|--------|-----------|
| `llm_additions` | LLM-generated | Not integrated into constraint matching or feedback display |
| `design_hints` | LLM-generated | Not referenced in any runtime code path |
| `sbu_type` | Deterministic (node_enricher) | Present in test fixtures but not in active filtering logic |
| `smiles` | Deterministic (RDKit) | Stored for reference but not queried at runtime |
| `molecular_weight` | Deterministic | Never referenced in screening code |
| `formula` | Deterministic | Minimal use (element extraction only) |
| `spin_state` | Deterministic (QMOF) | Loaded but not used in filtering or display |

### 6.4 Vocabulary at Runtime

The unified vocabulary (`unified_vocabulary.json`) is actively loaded by `constraint_utils.py` at runtime. It serves as the canonical vocabulary that Agent 2 uses to express functional group constraints. The vocabulary provides:

- **Alias resolution:** Maps variant tag names (e.g., `aromatic_ring`, `Aromatic`) to canonical forms (e.g., `Aromatic`)
- **Vocabulary validation:** Ensures Agent 2 only uses approved tags that exist in the database
- **Case-insensitive lookup:** All tags are normalized to lowercase before matching

This means the SMARTS-based functional group tags in the JSON records must align with the vocabulary's canonical vocabulary for the screening to work correctly. The `enrich_generic_tags` post-processing step (scripts 12/08 in the PORMAKE/QMOF pipelines) ensures this alignment by adding parent/abstract tags that match the vocabulary's hierarchy.

## 7. Known Limitations

### 7.1 hMOF Metal Node Characterization

The hMOF pipeline (`hmof_pipeline_v2.py`) does not run `metal_enricher.py` on hMOF records. As a result, the following `metal_node` fields are hardcoded to `None` for all 51,163 hMOF entries:

- `nuclearity`, `connectivity`, `geometry`, `sbu_type`, `oxidation_states`, `has_open_metal_sites`, `coordinating_groups`, `ligand_chemistry`

Only the `metals` list (element symbols extracted from node SMILES) is populated.

**Impact on screening:** The hMOF matchmaker (`hmof_matchmaker.py`) filters on `metals`, `topology`, `functional_groups`, and gas adsorption properties -- none of which require the missing metal_node fields. Therefore, this limitation does not affect hMOF screening results.

**Reason:** The hMOF source data (MOFX-DB JSONL) provides node SMILES, so `metal_enricher.py` could in principle be run. This was not done because (a) hMOF uses only 4 metals (Zn, Cu, V, Zr) with limited node diversity, and (b) the screening engine does not query these fields for hMOF. Running the enricher would add metadata but not change any screening outcomes.

### 7.2 `llm_additions` Field

The `llm_additions` field was designed to capture functional groups or structural motifs that SMARTS pattern matching might miss (e.g., "paddlewheel", "UiO-type"). The LLM is explicitly prompted to provide these during enrichment.

**Actual content across databases (after GPT-4o-mini re-enrichment, 2026-05-16):**
- PORMAKE: 756/867 non-empty (87.2%), 111 empty (e.g., `["phenanthroline"]`, `["indium_oxide", "dioxolane"]`)
- QMOF: ~2% non-empty (e.g., `["paddlewheel", "tetrahedral node"]`) — original enrichment, not re-run
- hMOF: 43,120/51,163 non-empty (84.3%), 4,414 empty, 10 errors (e.g., `["paddlewheel", "flexible", "breathing"]`)
- CoRE-MOF: 887/1,002 non-empty (88.6%), 114 empty (e.g., `["MIL-type", "rod-shaped SBU"]`)

**Impact:** This field is **not read by any runtime code path** in the LLM4MOF screening engine. It was generated as supplementary metadata but is not integrated into constraint matching or feedback generation. The screening pipeline relies exclusively on the deterministic SMARTS-based `functional_groups` field.

## 8. Anticipated Questions

### 8.1 How were the 105 SMARTS patterns validated?

The patterns were curated from four sources: (1) RDKit's built-in Functional_Group_Hierarchy, (2) the Daylight SMARTS theory manual, (3) the SMARTS-RX database, and (4) manual additions for MOF-relevant groups not covered by general-purpose libraries (e.g., carboxylate with aromatic bond encoding, paddlewheel motifs).

An independent cross-validation was performed using GPT-5.2 on 12 representative building blocks (6 edges: E1, E101, E118, E52, E22, E129; 6 nodes: N1, N100, N102, N10, N428, N200). Raw XYZ files were sent to GPT-5.2 without the pipeline's output, and the LLM's independent analysis was compared against the pipeline's results for formula, molecular weight, metals, ring count, rigidity, and functional group detection. Results are in `1_PORMAKE_pipeline/output_samples/_crossval_report.json`. The cross-validation script is `1_PORMAKE_pipeline/10b_crossval.py`.

**Limitation:** No systematic per-pattern false-positive/false-negative rate assessment was performed across the full dataset. The SMARTS patterns are structurally deterministic (a pattern either matches or it doesn't), but edge cases exist where the same substructure could be classified differently depending on chemical context.

### 8.2 Why 105 patterns? What drove the selection?

The pattern count reflects a balance between coverage and precision for MOF chemistry:
- **50+ functional groups** cover the dominant coordinating chemistries in MOFs (carboxylate, amine, azole, hydroxyl, etc.)
- **13 scaffolds** cover the most common aromatic backbones in MOF linkers
- **25+ heterocycles** cover nitrogen, sulfur, and oxygen heterocyclic rings commonly found in MOF linkers
- **13 substituents** cover small pendant groups that modify pore chemistry

Patterns are ordered from most specific to most general within each category to minimize over-matching. The set was iteratively refined during pipeline development: patterns were added when known functional groups went undetected, and removed or modified when false positives were observed.

### 8.3 Are the screening-critical fields reproducible?

Yes. All fields that drive the screening pipeline (see Section 5.1) are deterministic:
- `functional_groups`: Produced by RDKit SMARTS matching against a fixed pattern library. Given the same SMILES input and RDKit version, results are bit-for-bit identical.
- `metals`, `nuclearity`, `connectivity`, `is_rigid`, `length`: Computed from atomic coordinates and element data. Fully deterministic.
- `abstract_features`: Boolean flags derived from SMARTS matching. Deterministic.

LLM-generated fields (`readable_name`, `design_hints`) use temperature = 0.3 with JSON structured output, which produces near-deterministic but not guaranteed identical output across API versions. However, these fields are used only for display in the feedback loop and do not affect screening results.

### 8.4 What is the MOFid decomposition success rate?

MOFid decomposes each MOF's crystal structure into topology + node SMILES + linker SMILES. Success rates vary by database:

| Database | Node SMILES Valid | Linker SMILES Valid | Topology Assigned |
|----------|------------------|--------------------|--------------------|
| QMOF | 86.8% (17,678/20,373) | 86.1% (17,545/20,373) | 65.3% (13,305/20,373) |
| hMOF | ~95% | ~95% | 100% (75% are pcu) |
| CoRE-MOF | ~83% | ~99% | 59.3% (594/1,002) |

Records with missing SMILES or topology are retained with `status: "warning"`. All other fields (gas adsorption, bandgap, structural descriptors) remain valid. Downstream users can filter by `status: "pass"` for fully complete records.

### 8.5 How are gas adsorption units handled?

The hMOF pipeline preserves source units from the MOFX-DB API without normalization. The original GCMC simulation papers used different unit conventions:
- CH4: cm3(STP)/cm3 in source, stored as cm3(STP)/g in our index (likely converted during original data acquisition)
- CO2: mol/kg in source, stored as cm3(STP)/g
- H2: g/L in source
- Xe/Kr: mol/kg

For the PORMAKE benchmark CSVs, unit-converted variants are provided in three formats (cm3(STP)/cm3, mol/kg, g/L) to support different analysis needs. The `DATABASE_CONDITIONS.md` file in the main project documents all conditions, units, and source publications.

### 8.6 Why Gemini 2.0 Flash for PORMAKE but GPT-4o-mini for QMOF/hMOF?

The PORMAKE building block pipeline was developed first (March 2026) using Gemini 2.0 Flash via Google's `generativeai` Python SDK. When the QMOF and hMOF pipelines were built subsequently, GPT-4o-mini was used via the OpenAI API for its structured JSON output maturity and lower per-token cost at scale (72,000+ records). Both models use temperature = 0.3 and produce the same output schema (`readable_name`, `design_hints`, `llm_additions`). The LLM-generated fields are used only for display and do not affect screening results.

### 8.7 How was the aromatic bond encoding issue discovered and fixed?

PORMAKE XYZ files encode resonance-delocalized bonds (nitro, carboxylate, sulfonate, phosphonate) as bond type `A` (aromatic) in their explicit bond block. When RDKit constructs a molecule from these bonds, the resulting internal representation has aromatic C-O bonds rather than the conventional C=O / C-O resonance pair.

Standard SMARTS patterns for carboxylate (e.g., `[CX3](=O)[OX1-]`) specify explicit double bonds, which fail to match aromatic-encoded bonds. The fix uses **bond-order-agnostic** SMARTS:
- Carboxylate: `[#6X3](~[#8])(~[#8])` — matches carbon bonded to two oxygens regardless of bond order
- Nitro: `[#7]([#8])([#8])` — matches nitrogen bonded to two oxygens

This increased carboxylate detection from 2 to 202 building blocks. A subsequent **metal-stripping fallback** (removing metal atoms from SMILES before SMARTS matching) increased total enrichment from 667 to 808 of 867 building blocks.
